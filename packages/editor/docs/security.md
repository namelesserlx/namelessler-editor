# Security Model

The editor package provides client-side safety boundaries for content parsing and rendering. It does not replace server-side validation, authorization, CSRF protection, or upload security.

## XSS Boundaries

HTML is treated as untrusted input.

The package sanitizes:

- HTML passed to `importEditorContent(..., { from: 'html' })`
- HTML generated by `exportEditorContent(..., { to: 'html' })`
- pasted HTML inside the React editor
- readonly-rendered HTML

Sanitizing is based on DOMPurify plus package-level policy checks.

Default behavior:

- remove scripts
- remove event handler attributes such as `onclick`
- remove inline `style`
- remove dangerous URLs such as `javascript:`
- remove iframe unless explicitly enabled

## URL Policy

`sanitizeUrl` allows safe protocols by default:

- `http:`
- `https:`
- `mailto:`

Relative URLs are allowed by the default URL policy, but protocol-relative URLs are not. Data image URLs are disabled unless the caller explicitly enables them.

## Iframe Policy

Iframe embeds are disabled by default. Enabling iframe requires an explicit allowlist:

```ts
const htmlPolicy = {
  iframe: {
    enabled: true,
    allowedHosts: ['player.example'],
  },
};
```

Allowed iframe attributes are intentionally narrow:

- `src`
- `title`
- `allow`
- `allowfullscreen`
- `loading`

Iframe `src` must be HTTPS and its hostname must match `allowedHosts`.

The package does not ship provider presets. Applications own their provider choices.

## CSRF

The editor package does not call business APIs, so it does not manage CSRF tokens.

Applications and servers must protect state-changing endpoints with their normal API security model:

- authentication and authorization
- `SameSite` cookie settings
- Origin or Referer checks
- CSRF tokens where cookie-based auth needs them
- rate limiting for expensive endpoints
- server-side validation of all content and assets

## Uploads

Upload transport, upload UI adapters, asset contracts, and file validation are outside this package boundary.

Applications must validate uploads on both client and server:

- MIME type
- extension
- byte size
- image dimensions when relevant
- storage key/path ownership
- returned asset URL safety

Business-specific extensions such as `@tiptap-codeless/extension-file-upload` should be imported by the application and passed through the editor `extensions` prop.

## Server-Side Requirement

Client-side sanitizing improves UX and reduces accidental unsafe content, but it is not a trust boundary.

Before publishing or rendering user-controlled content, the server should still:

- validate that stored JSON matches the allowed schema
- sanitize any generated HTML used outside the editor package
- reject unsafe iframe providers
- reject unsafe URLs
- enforce authorization on edit and upload APIs
